## Goal

When the agent clicks a bookmark during a Live Guided Tour, the visitor's Matterport iframe should reload **without** the centered Title overlay and **without** Matterport's brand watermark. We achieve this by forcing `&title=0&brand=0` onto the rebuilt iframe URL, alongside the existing `&qs=1&play=1`.

## Where the URL is actually built

Bookmarks (`liveTourStops`) only carry `ss` (sweep id) and `sr` (rotation). The actual Matterport URL the visitor's iframe loads on teleport is constructed at runtime by a single helper in the generated end-product HTML:

`src/lib/portal.functions.ts` → `rewriteIframeForTeleport(baseUrl, ss, sr)` (lines ~2795–2808).

Today it:
1. Strips any pre-existing `ss`, `sr`, `qs`, `play` from the property's base `iframeUrl`.
2. Re-appends `ss=…&sr=…&qs=1&play=1`.

The visitor calls this from `applyTeleport()` (line 2814) when a teleport packet arrives over the WebRTC DataChannel. This is the **only** code path that mutates the iframe `src` during a live session — confirmed by ripgrep across `portal.functions.ts` and `live-session.mjs`.

## Why this is the safest single point of change

- The bookmark data model (`LiveTourStop` in `src/components/portal/types.ts`) and wire format (`{type:"teleport", ss, sr}` in `live-session.mjs`) deliberately store **only** `ss`/`sr` — no full URL. Adding params to stored data would bloat the model and require a migration.
- The base `iframeUrl` per property is generated by `buildMatterportUrlServer` from the user-controlled `TourBehavior` (which already exposes `hideTitle` and `hideBranding` toggles). We must NOT permanently mutate that base URL because:
  - A presentation may legitimately use branding/title outside live tours.
  - The non-live (normal) viewing experience must keep respecting the agent's `TourBehavior` settings.
- Forcing the params **only inside `rewriteIframeForTeleport`** scopes the override to the exact moment the visitor is teleported via a bookmark — the only situation the user described.

## Implementation

Single, surgical edit in `src/lib/portal.functions.ts` inside `rewriteIframeForTeleport`:

1. Extend the strip regex to also remove pre-existing `title` and `brand` query params, so we can guarantee our values win regardless of what `iframeUrl` already contains:
   - Change `/[?&](ss|sr|qs|play)=[^&]*/g` → `/[?&](ss|sr|qs|play|title|brand)=[^&]*/g`.
2. Append `&title=0&brand=0` to the rebuilt query string, right after `&qs=1&play=1`:
   - `qs += "&qs=1&play=1&title=0&brand=0";`
3. Update the inline comment above the function so future maintainers know live-tour teleports always force a clean (no-title, no-brand) reload.

That's it. No type changes, no schema changes, no UI changes, no changes to `live-session.mjs`, `HudPreview.tsx`, or the `TourBehavior` model.

## Ripple-effect audit

- **Normal (non-live) tour viewing**: untouched. `iframeUrl` is set straight from `props[i].iframeUrl` at lines 1445, 1559, 2652, 2687 — none of these go through `rewriteIframeForTeleport`. The agent's `hideTitle`/`hideBranding` behavior toggles still drive that URL exactly as before.
- **Builder preview / `HudPreview.tsx`**: doesn't call `rewriteIframeForTeleport` (it's runtime-only inside the generated HTML).
- **Initial visitor load before any teleport**: unchanged — uses `iframeUrl` directly.
- **Re-teleport to the same stop**: still works; the strip-and-rebuild is idempotent and now also strips any leftover `title`/`brand` from a previous teleport.
- **Saved bookmarks**: unchanged. We never persist or transmit `title`/`brand`; only the runtime URL builder injects them.
- **Tests**: no test currently asserts the exact query string of `rewriteIframeForTeleport` (verified via `rg` of `tests/`). Existing `tests/live-session.test.mjs` covers the wire protocol, not URL rewriting, so it's unaffected.
