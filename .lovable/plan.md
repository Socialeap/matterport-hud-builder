## Review verdict

All 4 proposals are sound and worth adopting. One refinement before we build:

**Custom Q&As (#3) need their own storage, not the existing `canonical_qas` JSONB.**
`property_extractions.canonical_qas` is auto-regenerated by the extractor on every re-run — agent-authored answers would get wiped. We need a separate `custom_qas` table (or a `custom_qas` JSONB column on `saved_models`) so human-authored Q&As survive re-extraction. I recommend a dedicated table, keyed by `saved_model_id` + `property_uuid`, with RLS scoped to the provider.

Everything else in the proposal is adoptable as written.

## Implementation plan

### Phase 1 — Intelligence Review dashboard (proposal #1)

- Add `IntelligenceReviewPanel` component, opened from `PropertyIntelligenceSection` as a modal.
- Reads `candidate_fields` JSONB from `property_extractions` for the active property.
- Renders one card per candidate: fact (key + value), confidence score, evidence snippet (resolved from `field_provenance` → source chunk).
- Three actions per card:
  - **Approve** → move entry from `candidate_fields` into `fields`, drop from candidates.
  - **Edit** → inline editor that writes the corrected value into `fields`, drops from candidates.
  - **Discard** → remove from `candidate_fields` only.
- All writes go through a new `reviewCandidate` server function (`createServerFn` + `requireSupabaseAuth`) that updates the row atomically and re-runs the canonical-QA + intelligence-health builders so downstream consumers stay in sync.
- After any mutation, invalidate the property indexing cache so the runtime picks up the curated facts.

### Phase 2 — Hyper-grounded synthesis prompt (proposal #2)

- Edit `SYSTEM_PROMPT` in `supabase/functions/synthesize-answer/index.ts`:
  - Strict-context rule ("ONLY answer based on the provided JSON and Document Chunks").
  - No-meta-language rule (no "Based on the documents…", "It appears…").
  - Refusal template that interpolates the agent's name/contact when context is missing.
- Pass `agent_name` + `agent_contact` (already in the property brain) into the synthesis request body and weave them into the refusal sentence template.
- Lower temperature from 0.3 → 0.1 to reduce drift.
- Update the Groq fallback path to use the same prompt.

### Phase 3 — Custom Q&A layer (proposal #3, refined)

- **Schema**: new `custom_qas` table — columns: `provider_id`, `saved_model_id`, `property_uuid`, `question`, `answer`, `embedding vector(384)`, `created_at/updated_at`. RLS: providers manage their own rows; bound clients can read.
- **UI**: "Custom Q&A" manager inside `PropertyIntelligenceSection` — list, add, edit, delete. Embedding is generated client-side via the existing embedding worker on save.
- **Build pipeline**: include `custom_qas` rows in the property brain alongside `canonical_qas`, exported to `window.__PROPERTY_EXTRACTIONS__` in the standalone HTML so the runtime stays self-contained (per memory: end product must not phone home).
- **Runtime**: in `ask-runtime-logic.mjs`, before any retrieval/synthesis:
  1. Exact-match check (normalized) against custom Q&As → return immediately.
  2. Cosine similarity ≥ 0.92 against custom Q&A embeddings → return immediately, bypass Gemini.
  3. Otherwise fall through to existing canonical → retrieval → synthesis pipeline.
- Decrement-free: instant-answer hits should skip the ask quota counter.

### Phase 4 — Retrieval weight tuning (proposal #4)

- In `ask-runtime-logic.mjs`:
  - Boost field-derived chunks (`kind === "field_chunk"`) score by a configurable multiplier (start at 1.5×) before merging with raw-chunk hits.
  - Add a keyword-pin step: tokenize the query, intersect against the set of field keys (HVAC, Roof, HOA, gate code, etc.) using the canonical-keys synonym map, and force-include matching field chunks at the top of the trusted set even when their semantic score is below threshold.
- Keep `hybridSearchFor` in `orama-search.ts` API-compatible; the weighting happens in the consumer so we don't fork Orama internals.
- Add a unit test in `tests/` that asserts a "what's the HOA fee" query surfaces the HOA field chunk over a marketing prose chunk.

### Cross-cutting

- New migration: `custom_qas` table + RLS + trigger to bump `updated_at`.
- Update `property-brain.mjs` and the standalone HTML exporter (`protected-export.ts`) to embed `custom_qas` and the curated `fields` snapshot.
- Update `intelligence-health` to count "human-approved" facts separately and surface that in `IndexingStatusBadge`.
- Tests: extend `tests/ask-pipeline-integrity.test.mjs` for the instant-answer short-circuit and the field-boost behavior.

## Risk & rollout

- Phase 2 is the lowest-risk, highest-impact change — ship first.
- Phase 4 ships next (pure runtime, no schema).
- Phase 1 + 3 ship together since they share the review UI shell and both need the migration.
- All four phases preserve the "self-contained .html" invariant — no phone-home introduced.
