

## Why both URLs render the same page

`p.$slug.tsx` and `p.$slug.demo.tsx` form a **parent → child** route pair in TanStack's flat-file routing. When the URL is `/p/transcendencemedia/demo`, TanStack matches:

1. `/p/$slug` (parent) → renders `HudBuilderSandbox` (the builder)
2. `/p/$slug/demo` (child) → would render inside the parent's `<Outlet/>`

But `p.$slug.tsx` does **not render `<Outlet/>`** — it just returns `<HudBuilderSandbox />` with the optional banner. So the child route is "matched but never mounted." The browser shows only the parent (builder) for *both* URLs. Confirmed via screenshot of the live site: `/demo` shows "Build Your 3D Presentation" with the Branding/Property Models form and the blue "View Demo" banner at the top — that banner only exists inside `p.$slug.tsx`, proving the parent is what's rendering.

## Fix: convert `/p/$slug` into a true layout split

Use TanStack's standard layout + index pattern so the two URLs render independent leaf components:

```text
src/routes/
  p.$slug.tsx          ← becomes a layout — only renders <Outlet />
  p.$slug.index.tsx    ← NEW — moves the builder loader + HudBuilderSandbox here (handles "/p/:slug")
  p.$slug.demo.tsx     ← unchanged — handles "/p/:slug/demo"
```

### Changes

**1. Create `src/routes/p.$slug.index.tsx`** (new)
- Move the entire current contents of `p.$slug.tsx` here (loader, `fetchBrandingBySlug`, `checkDemoPublished`, `PortalPage` component, the blue "View Demo" banner, `<HudBuilderSandbox />`).
- Change the route path to `createFileRoute("/p/$slug/")` (trailing slash = index route).
- Keep the notFoundComponent for missing branding.

**2. Rewrite `src/routes/p.$slug.tsx`** (becomes pure layout)
- Strip out the loader and builder rendering.
- Keep only:
  ```tsx
  import { createFileRoute, Outlet } from "@tanstack/react-router";
  export const Route = createFileRoute("/p/$slug")({
    component: () => <Outlet />,
  });
  ```
- No loader here — each child route owns its own data fetch. This avoids double-fetching on `/demo`.

**3. `src/routes/p.$slug.demo.tsx`** — no code change needed
- It already declares `createFileRoute("/p/$slug/demo")` and renders the full-viewport `HudPreview`. Once the parent stops swallowing the child, this leaf will finally mount.

**4. `src/routeTree.gen.ts`** — auto-regenerated by Vite plugin on build. Do not touch.

### Result

| URL | Renders |
|---|---|
| `/p/transcendencemedia` | `p.$slug.tsx` layout → `<Outlet>` → `p.$slug.index.tsx` → builder (`HudBuilderSandbox`) with "View Demo" banner |
| `/p/transcendencemedia/demo` | `p.$slug.tsx` layout → `<Outlet>` → `p.$slug.demo.tsx` → full-viewport `HudPreview` with bottom CTA strip |

Two genuinely distinct pages — same as the dashboard demo preview vs. the live presentation experience.

### Ripple safety trace

| Touched | Used elsewhere | Risk | Mitigation |
|---|---|---|---|
| `p.$slug.tsx` rewrite to layout | Only renders for `/p/$slug/*` URLs | None — content moves to index file | All loader/component logic preserved verbatim in `p.$slug.index.tsx` |
| New `p.$slug.index.tsx` | Renders for `/p/$slug` exactly | None — net-new file with copied logic | — |
| `p.$slug.demo.tsx` | `/p/$slug/demo` | None — untouched | — |
| `<Link to="/p/$slug">` (in demo CTA strip) | TanStack flat routing still resolves to `/p/$slug` index route correctly | None | — |
| `<Link to="/p/$slug/demo">` (in `p.$slug.index.tsx` banner) | Resolves to demo leaf | None | — |
| `routeTree.gen.ts` | Auto-generated | Don't edit | Vite plugin regenerates on dev/build |
| `checkDemoPublished` server fn | Called from index loader only | None | Same call, same result |
| Any custom domain routing (Pro tier) | Path-based routing unaffected | None | — |

### Out of scope

- No change to publish flow, license check, branding, or `getPublicDemoBySlug`.
- No change to dashboard sandbox preview (`/dashboard/demo`).
- No URL changes — `/p/{slug}` and `/p/{slug}/demo` keep identical public URLs, so existing shared links keep working.
- No UI redesign of either page — only the routing wiring changes; the demo page already has its full-viewport HudPreview design ready to display.
